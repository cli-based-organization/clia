## Résultats

- FND : le LLM Wiki (post-RAG) stocke la connaissance en markdown curé lu dans le contexte, maintenu par le LLM ; l'OKF de Google (v0.1) le standardise en un répertoire de markdown à frontmatter YAML dont le seul champ requis est `type`, les liens formant un graphe ; OKF vise l'interop **structurelle** et laisse la sémantique (ontologies RDF/OWL) aux conventions.
- ANL : **`clia` est déjà une instance du pattern LLM Wiki et dépasse OKF sémantiquement** (typologie `ADR-004`, skills-spécifications, versionnage, validation). Adopter l'OKF **totalement** serait une régression ; l'adopter **sélectivement** (frontmatter `type` uniforme + champs réservés, cross-références en liens markdown) apporte de la **portabilité inter-agents** et de l'outillage sans perte de sémantique. Impact : amender tous les skills (frontmatter OKF), migrer l'existant, arbitrer le recouvrement `ressources.yaml`/logs vs `index.md`/`log.md`, et trancher le mapping `clia ↔ OKF` — travail à intégrer à la **réécriture d'`ADR-004`** déjà prévue.
- Recommandations : OKF comme couche d'interop structurelle (pas comme modèle sémantique) ; uniformiser les métadonnées en frontmatter YAML (remplacer les puces, pas les doubler, pour respecter le principe de source unique) ; formaliser les cross-refs en liens markdown ; conserver `ressources.yaml`/logs et mapper vers OKF à l'export ; adopter par étapes vu la v0.1 ; trancher par ADR lié à `ADR-004`.
