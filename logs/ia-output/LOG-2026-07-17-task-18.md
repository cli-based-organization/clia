## Résultats

- FND exhaustive sur MCP : protocole ouvert (JSON-RPC 2.0), architecture hôte/client/serveur, primitives tools/resources/prompts, transports stdio/Streamable HTTP, mise en place côté Claude (Desktop, Code, claude.ai, API, Managed Agents), rapport intime aux CLI, et surface de sécurité (tool poisoning, prompt injection) exigeant HITL et OAuth 2.1.
- ANL critique : **l'intérêt de MCP pour `clia` n'est pas activé aujourd'hui**. Bénéfices (portabilité inter-clients, accès structuré typé) sans besoin actuel (mono-dépôt, agent unique Claude Code, accès déjà découvrable via Bash + fichiers) ; coûts (runtime/serveur, JSON-RPC, sécurité) heurtant l'identité déterministe et minimale de `clia` (`SPEC-001`/`REQ-001`) et la séparation `ADR-007`.
- Recommandations : ne pas introduire MCP maintenant ; si besoin de structure, ajouter plutôt une sortie `--json` aux commandes d'inspection (coût moindre, cohérent bash) ; rouvrir la question MCP seulement si `clia` doit servir des clients non-Claude-Code, exposer une inspection typée inter-dépôts, ou imposer un gating de permission uniforme ; si un jour retenu, n'exposer que la lecture seule via un serveur mince, avec garde-fous de sécurité.
