## Résultats

- Deux FND business rapatriées (point fixe, non versionnées).
- FND exhaustive sur requis/spécification : le champ est terminologiquement surchargé ; la majorité des références (Zave & Jackson, usage courant) traitent le requis comme le besoin (amont) et la spécification comme la description d'interface (aval, plus concrète) ; la lecture inverse (spécification = quoi abstrait) est minoritaire ; l'opposition quoi/comment n'a de sens qu'une fois la frontière machine fixée.
- ANL comparée : la divergence signalée est réelle — `clia` (REQ besoin → SPEC interface) et `ticket-driven-ai` (SPC quoi abstrait → REQ comment concret) **inversent** l'axe d'abstraction de « spec » et « req ». `clia` est mieux aligné sur la littérature dominante ; `ticket-driven-ai` est cohérent mais nomme « requis » ce que le champ appelle conception/contraintes. Défauts relevés : duplication verbatim d'ADR dans `ticket-driven-ai` (002≡005, 003≡006, 004≡007) ; frontière machine non explicitée dans les deux dépôts ; modèles de livrable non interopérables (promotion de zone vs cycle-de-vie+versionnage).
- Recommandations : figer la convention SPEC/REQ de `clia` dans un ADR citant la FND ; expliciter la frontière machine ; assumer que REQ porte des requis-contraintes et que SPEC est une spec de conception ; aligner l'écosystème sur la convention `clia` ; signaler la duplication d'ADR de `ticket-driven-ai` ; réinterpréter les définitions importées dans le modèle de livrable de `clia`.
